1 Introduction & Scope

VibeBoard is a lightweight, single-page project dashboard that helps a solo developer or small team manage multiple software projects simultaneously. The core problem it solves: when you can only physically work on one project at a time, how do you keep track of where you left off in the others?

This document describes the system from the user's perspective. Every feature is expressed as a user story with acceptance criteria that define when the story is done. These stories are the source of truth for the development backlog — each one maps to one or more tasks in the tracker.

Scope: v3.0 covers multi-project container management, task CRUD with status workflow, worker allocation and tracking, dashboard metrics and visualisation, park/resume lifecycle, real-time data refresh, and agent orchestration with pipeline tracking and swarm visibility. Authentication, user management, and API integrations are out of scope for this version.

2 Glossary

Project
A named container that holds its own tasks, workers, and metrics. Only one project can be active at a time. Projects can be active, parked (with a save-state note), archived, or deleted.
Task
A single unit of work belonging to a project. Moves through a lifecycle: Backlog → In Progress → QA-Review → Done. Can be assigned to a worker.
Worker
A person or software agent that can be assigned to tasks. Has a role (e.g. Frontend Developer, QA Engineer) and a status (idle or busy).
Park / Park Note
Suspending work on the current project. The system saves an auto-generated summary (tasks done, in-progress, which workers were on what) plus any manual note the user adds, so context is preserved when the project is reactivated later.
Current Project
The single project whose tasks, workers, and metrics are shown on the dashboard. Switching the current project re-scopes every view.
Status (Task)
One of: Backlog (not started), In Progress (actively being worked), QA-Review (awaiting verification), Done (complete).
Auto-Switch
When the current project is parked or deleted, the system automatically selects the next available active project so the dashboard never shows an empty state.

3 User Personas

Persona: Solo Developer
A single developer (possibly with AI agent workers) managing multiple small-to-medium projects. Juggles context between them. Needs to park one project and resume another without losing their place. Acts as project manager, developer, and QA simultaneously.
Persona: Small Team Lead
Manages 1–3 developers and a QA person across one active project. Allocates tasks, checks progress, re-assigns blocked work. Uses the dashboard for stand-ups and status checks.
Persona: AI Agent Worker
A software agent (like Grunt1, Pikey, SpecBot) assigned to tasks. Has a defined role, can be busy or idle, and the dashboard shows what task they're currently on. Agents do not log in — they are tracked by the dashboard.
Primary persona: Solo Developer. The system is optimised for one-person-multiple-projects use. Team-lead features (e.g. multi-user login, permissions) are future scope.

4 Feature: Project Container

Projects are the top-level organisational unit. Every task, worker allocation, and metric lives inside exactly one project. The user sees exactly one project at a time.

US-01: Create a new project

User Story
As a Solo Developer, I want to create a new project with a name and optional description so that I can begin organising work for a different codebase or initiative.
Acceptance Criteria
  • A "New Project" form is accessible from the Manage Projects modal
  • The form requires a name (1–255 characters) and accepts an optional description
  • Submitting the form creates the project with status "active"
  • After creation, the new project becomes the current project — the dashboard instantly re-scopes to show it
  • The new project appears in the project selector dropdown and in the Manage Projects list
  • If the API fails, an error toast is shown and the dashboard state is unchanged

US-02: Switch between projects

User Story
As a Solo Developer, I want to switch which project I'm looking at from a single click in the header so I can quickly context-switch between my projects without leaving the dashboard.
Acceptance Criteria
  • A dropdown in the header shows all projects in order: active first, then parked, then archived
  • Each dropdown entry shows the project name plus task count (e.g. "VibeBoard 17 tasks, 2 done")
  • The currently active project is visually indicated (checked or highlighted)
  • Clicking a project in the dropdown immediately makes it the current project
  • After switching, every dashboard section (stats, chart, workers, task table) updates to show that project's data
  • Switching does not change the status of the project being left
  • The page title updates to reflect the current project name

US-03: See current project details at a glance

User Story
As a Small Team Lead, I want the current project's name, description, and summary stats visible in the header and dashboard so I always know what context I'm working in.
Acceptance Criteria
  • The header displays the current project name next to a dropdown trigger
  • The db-status indicator shows connection health
  • The stats cards (total, completed, in-progress, backlog, progress %) reflect the current project only
  • Every data-fetching API call is scoped to the current project's ID

US-04: Delete a project

User Story
As a Solo Developer, I want to delete a project I no longer need so that my project list stays clean and uncluttered.
Acceptance Criteria
  • A delete button appears on each project row in the Manage Projects modal
  • Clicking delete shows a confirmation prompt ("Are you sure you want to delete [project name]?")
  • On confirm, the project is deleted from the database and its tasks are unlinked (project_id set to NULL, not cascade-deleted)
  • If the deleted project was the current project, the system auto-switches to the next available project
  • A success toast is shown; the project vanishes from dropdown and manage list

US-05: Manage all projects from one modal

User Story
As a Small Team Lead, I want a single "Manage Projects" view where I can see all projects side by side with their status, task counts, and action buttons so I can maintain an overview of everything on my plate.
Acceptance Criteria
  • "Manage Projects" is accessible from the bottom of the project selector dropdown
  • The modal lists every project in a table with columns: Name, Status, Tasks, Progress %, Park Note, Actions
  • Action buttons per project: Activate (makes it current), Park (opens park modal), Delete (with confirmation)
  • A "New Project" button at the top of the modal opens an inline creation form
  • Closing and re-opening the modal refreshes the data

5 Feature: Task Management

Tasks are the atomic unit of work. Each task belongs to exactly one project and optionally one worker. Tasks flow through a defined status lifecycle.

US-06: View tasks for the current project

User Story
As a Solo Developer, I want to see all tasks for my current project in a sortable table so I know what needs to be done and what's in progress.
Acceptance Criteria
  • The task table displays ID, Title, Status (coloured badge), Worker (name or "—"), Created date, and Actions (Edit, Delete)
  • Only tasks belonging to the current project are shown
  • Switching projects re-populates the table with that project's tasks
  • Long task lists support scrolling within the table container

US-07: Create a new task

User Story
As a Solo Developer, I want to add a task to the current project by filling in a title, description, and optionally setting its status and worker, so I can capture work items as they come up.
Acceptance Criteria
  • A "New Task" button opens a modal form with fields: Title (required), Description (optional, textarea), Status (dropdown, default "Backlog"), Worker (dropdown of idle+busy workers or unassigned)
  • Submitting creates the task scoped to the current project
  • The new task appears instantly in the table without a page reload
  • If a worker is assigned, the worker card updates to show them as busy with the new task
  • The stats counters update to reflect the new task

US-08: Edit a task

User Story
As a Solo Developer, I want to change a task's title, description, status, or worker assignment so I can correct mistakes and track progress.
Acceptance Criteria
  • An "Edit" button on each task row opens a pre-filled modal with the task's current values
  • Changing status updates the badge colour in the table immediately
  • Changing worker assignment updates worker cards (old worker becomes idle if no other task, new worker becomes busy)
  • Switching a task to "Done" increments the completed counter and recalculates progress %

US-09: Delete a task

User Story
As a Solo Developer, I want to remove a task that was added by mistake or is no longer relevant, so the backlog stays accurate.
Acceptance Criteria
  • A "Delete" button on each task row removes it after confirmation
  • The task vanishes from the table and the stats are recalculated
  • If the task was assigned to a worker, that worker's card updates (becomes idle if no other active task)

US-10: Move a task through its status lifecycle

User Story
As a Small Team Lead, I want a clear status workflow so the team has a shared understanding of where each task is — from idea (Backlog) to active work (In Progress) to verification (QA-Review) to completion (Done).
Acceptance Criteria
  • Valid statuses are: Backlog, In Progress, QA-Review, Done
  • Each status has a distinct badge colour: grey (Backlog), blue (In Progress), orange (QA-Review), green (Done)
  • The status chart (doughnut) updates when any task moves between statuses
  • Progress % is calculated as: (task count where status=Done) / (total task count) * 100

US-11: Task count and progress scoped to project

User Story
As a Solo Developer, I want the task counts and progress percentage to reflect only the current project, so I'm not misled by stale numbers from another project.
Acceptance Criteria
  • Every task-related API endpoint accepts or derives the current project scope internally
  • Switching projects recalculates all metrics from zero for the new project
  • A project with zero tasks shows 0% progress, not an error

6 Feature: Worker Allocation

Workers represent people or agents that do the work. The dashboard tracks who is busy, who is idle, and what they're working on — scoped to the current project.

US-12: See all workers and their current task

User Story
As a Small Team Lead, I want to see all workers in the system with their role, status (busy/idle), and which task they're currently assigned to so I can allocate work effectively during stand-ups.
Acceptance Criteria
  • Worker cards are displayed in a dedicated section below the stats
  • Each card shows: initials avatar, name, role, status badge (BUSY/IDLE), and the title of their current task (or "No task assigned")
  • The section header shows "X active / Y total" where active = workers with tasks in the current project
  • Worker data is scoped: only tasks within the current project determine busy/idle status

US-13: Assign a worker to a task

User Story
As a Solo Developer, I want to assign a worker to a task when creating or editing it, so responsibility is clear and the dashboard reflects who is doing what.
Acceptance Criteria
  • The task create/edit modal includes a "Worker" dropdown listing all workers
  • Selecting a worker and saving immediately updates that worker's card to show the new task
  • A worker can be assigned to multiple tasks — they are "busy" if they have at least one non-Done task
  • Unassigning a worker from all tasks returns them to "idle"

US-14: Dashboard updates when worker status changes

User Story
As a Small Team Lead, I want the worker section to refresh automatically so I don't have to manually poll for changes when tasks are re-assigned.
Acceptance Criteria
  • Worker status refreshes on every dashboard auto-refresh cycle (30 seconds)
  • Manual refresh button also reloads worker data
  • Worker cards update within one refresh cycle of a task being assigned or completed

7 Feature: Dashboard & Metrics

The dashboard is the primary interface. It gives an at-a-glance view of project health, task distribution, and worker allocation.

US-15: See project-level summary statistics

User Story
As a Solo Developer, I want to see five key numbers at the top of the dashboard — Total Tasks, Completed, In Progress, Backlog, and Progress % — so I can assess project health at a glance.
Acceptance Criteria
  • Five stat cards are displayed in a row at the top of the main content area
  • Each card has an icon, the numeric value, and a label
  • Progress % is computed as (done / total) × 100, shown as an integer percentage
  • All stats update when the project switches or tasks change

US-16: Visualise task distribution with a chart

User Story
As a Small Team Lead, I want a doughnut chart showing the status breakdown so I can quickly see whether the team is balanced across active work vs backlog.
Acceptance Criteria
  • A Chart.js doughnut chart appears below the stats section
  • The chart segments are colour-coded by status: Backlog (grey), In Progress (blue), QA-Review (orange), Done (green)
  • The chart title reads "Task Distribution" and is scoped to the current project
  • Switching projects re-renders the chart with the new project's data

US-17: Auto-refresh dashboard data

User Story
As a Solo Developer, I want the dashboard to refresh itself automatically every 30 seconds so I always see the latest state without clicking anything.
Acceptance Criteria
  • An auto-refresh timer fetches all dashboard data every 30 seconds
  • Auto-refresh does not cause a full page reload — data updates in place
  • A manual "Refresh" button is available for immediate updates
  • Auto-refresh is paused while a modal is open to avoid disrupting the user
  • If an API call fails during auto-refresh, a toast shows the error but the existing data remains visible

US-18: Visual feedback during data loading

User Story
As a Solo Developer, I want a loading spinner when data is being fetched so I know the system is working and hasn't frozen.
Acceptance Criteria
  • A full-page loading overlay with a spinner appears during initial data load
  • Subsequent auto-refreshes do not show the overlay (data updates silently)
  • Failed API calls show an error toast notification, not a spinner that never resolves
  • Toast notifications auto-dismiss after 4 seconds

8 Feature: Park & Resume

The park/resume lifecycle is the heart of the multi-project workflow. It lets the user suspend work on one project with full context and pick up another.

US-19: Park the current project with context

User Story
As a Solo Developer, I want to park the current project with a single click and have the system automatically save a summary of where I left off, so I can come back weeks later and know exactly what was happening.
Acceptance Criteria
  • A "Park" button is always visible in the header nav bar
  • Clicking "Park" opens a modal showing an auto-generated summary (e.g. "2 done, 2 in progress (12% complete). Grunt1 on 'Build API endpoints', Pikey on 'Integration Test Update'")
  • The user can add a manual note in a textarea on the park modal
  • Confirming the park sets the project's status to "parked" and saves the park note to the database
  • After parking, the system auto-switches to the next available active project
  • The parked project appears in the dropdown with a "parked" indicator

US-20: Resume (reactivate) a parked project

User Story
As a Solo Developer, I want to reactivate a parked project from the dropdown or Manage Projects modal so I can pick up exactly where I left off.
Acceptance Criteria
  • Clicking a parked project in the dropdown activates it and switches the dashboard view
  • Alternatively, the Manage Projects modal has an "Activate" button for each parked project
  • Reactivating sets the project status back to "active"
  • The park note is preserved in the database (not cleared on reactivate) so the user can refer to it later
  • The dashboard re-scopes to show the reactivated project's current data

US-21: See park notes in project list

User Story
As a Solo Developer, I want to see the park note for each parked project in the Manage Projects modal so I can decide which one to resume without having to open them one at a time.
Acceptance Criteria
  • The Manage Projects table shows a "Park Note" column for each project
  • Parked projects display their park note (truncated if longer than 80 characters)
  • Active and archived projects show "—" in the park note column

9 Non-Functional Requirements

NFR-01: Performance
The dashboard should feel instant. All API responses must complete within 200ms for typical project sizes (up to 50 tasks, 10 workers).
Acceptance Criteria
  • API response time for /api/tasks, /api/metrics, /api/workers stays under 100ms at P95 for typical data sizes
  • Full dashboard data load (5 concurrent API calls) completes in under 500ms
  • Auto-refresh does not cause visible UI stutter or flicker
NFR-02: Visual Consistency
The entire application uses a dark theme. All UI elements must be legible and consistent in styling.
Acceptance Criteria
  • Background colour is #0d1117 (GitHub dark) or equivalent
  • All modals, buttons, tables, and cards share consistent border-radius (8px), padding, and font sizing
  • Hover and active states are defined for all interactive elements
  • Status badges have distinct, semantic colours
NFR-03: Reliability
The system must handle common edge cases gracefully without showing raw errors to the user.
Acceptance Criteria
  • If the database is unreachable, the API returns a 503 with a JSON error message
  • The frontend shows a user-friendly error toast, not a blank page or raw JSON
  • Creating a project with a duplicate name is handled (unique constraint shows error toast)
  • Deleting the last remaining project: the Manage Projects modal shows "No projects" state
  • A project with zero tasks shows 0% progress and empty chart — no errors
  • All confirmations (delete task, delete project) require explicit user confirmation
NFR-04: Data Persistence
Park notes, task statuses, and worker assignments must survive server restarts.
Acceptance Criteria
  • All data is stored in MariaDB (not in-memory or session)
  • Stopping and restarting the PHP dev server preserves all data
  • Park notes written to the database are immediately readable on the next dashboard load

10 Feature: Agent Orchestration

VibeBoard is the control plane for a multi-agent software development swarm. The Brain (Orchestrator) decomposes user requests into atomic sub-tasks and dispatches them to named Worker Agents. Each worker has a defined role, toolset, and responsibility. VibeBoard tracks every task from creation through assignment, execution, and delivery — providing full visibility into the swarm's activity.

Key Concepts:
Brain — the orchestrator that plans, delegates, validates, and synthesises. Runs reasoning-heavy models.
Worker — a stateless execution agent that receives full context with every dispatch. Runs coding/utility models.
Pipeline — a sequential workflow (e.g. Spec → Plan → Code → Test) where each stage depends on the prior.
Swarm — the collection of all active workers. Workers run in parallel when independent, serial when dependent.
Fallback Chain — if the primary model fails (429/5xx/timeout), the system escalates through a ranked list of alternatives before reporting failure.
US-22: Track named worker agents
As a Brain orchestrator, I want to define named worker agents (Pikey, Grunt1, Grunt2, SpecBot, TesterBot) with their role, model, and toolset so that tasks can be assigned to the right agent for the job.
Acceptance Criteria
  • Workers have a name, role description, and status (Idle/Busy/Offline)
  • Each worker can be assigned a preferred model and provider
  • Worker list persists in the database and is editable via the Manage Agents modal
  • Worker allocation dashboard shows which worker is doing what in real time
US-23: Assign tasks to a named agent
As a Brain orchestrator, I want to assign any backlog task to a specific worker agent so that the swarm can execute work in parallel without conflict.
Acceptance Criteria
  • Task create/edit form includes an agent dropdown (selectable from the worker pool)
  • Assigning an agent changes the worker's status to Busy
  • When a task is moved to Done, the worker's status returns to Idle
  • Each worker can only have one active In Progress task at a time
US-24: View task execution pipeline
As a Brain orchestrator, I want to see the pipeline stages (Plan → Spec → Assess → Code → Test → Review → Done) for each task so that I know exactly where work is blocked or waiting.
Acceptance Criteria
  • Tasks can be assigned a pipeline stage in addition to their status
  • Pipeline stages are ordered and visible as a progress bar or step indicator
  • The dashboard can filter tasks by pipeline stage
  • Dependent tasks show which task(s) they are blocked by
US-25: Define task dependencies
As a Brain orchestrator, I want to mark tasks as blocked-by or depends-on other tasks so that the swarm executes work in the correct order without starting a task before its prerequisites are done.
Acceptance Criteria
  • Task detail view has a "Depends On" field referencing one or more task IDs
  • A task with unmet dependencies shows "Blocked" status and cannot be moved to In Progress
  • When a dependency task is completed, the blocked task auto-unblocks (status returns to Backlog/Ready)
  • Circular dependency detection — creating a cycle shows an error and is rejected
US-26: Log task execution results
As a Brain orchestrator, I want every delegated task to log its outcome — files created, test results, errors — so that I can audit what the swarm produced without re-running it.
Acceptance Criteria
  • Task detail page has an "Execution Log" section showing timestamps, worker output, and exit codes
  • Failed tasks show the error message and retry count
  • The log is append-only and preserved across deployments
  • Each log entry records: worker name, timestamp, duration, exit status, and summary
US-27: Automatic retry on failure
As a Brain orchestrator, I want failed tasks to automatically retry up to a configurable limit so that transient failures (rate limits, timeouts) don't require manual intervention.
Acceptance Criteria
  • Each task has a retry count and max retries setting (default: 3)
  • On failure, the task is re-queued with exponential backoff (2s, 4s, 8s delay)
  • After exhausting retries, the task is marked Failed and the Brain is notified
  • Retries use the next model in the fallback chain (not the same model that failed)
US-28: Swarm activity dashboard
As a Brain orchestrator, I want a live swarm view showing all active workers, their current task, duration, and model, so that I can monitor swarm health and spot stuck tasks at a glance.
Acceptance Criteria
  • Swarm view shows each worker as a card with: name, role, model, elapsed time, task title, status
  • Idle workers show "Waiting for assignment" with no elapsed time
  • Workers stuck on a task for more than 5 minutes are highlighted in amber
  • Clicking a worker card opens their current task detail
  • The view auto-refreshes every 10 seconds

11 Future Considerations

The following items are explicitly out of scope for v4.0. They are recorded here to inform future work without derailing current development.

FS-01
User authentication and multi-user access control. Currently the system assumes a single trusted user.
FS-02
Advanced task filtering and free-text search within the task table.
FS-03
Drag-and-drop task status changes (Kanban board view).
FS-04
Project-level settings or customisation (e.g. custom statuses, colour themes).
FS-05
API integrations with external tools (GitHub, Jira, Slack).
FS-06
Time tracking or deadline/overdue calculations per-task.
FS-07
Export/import of projects or task data.
